# Feature Analysis

TMDB 탐색 웹앱의 요구사항분석 절차. `analyst` 에이전트가 이 절차를 따른다. 이 스킬은 "이 요구사항이 뭐고 타당한가"만 다룬다 — "어떻게 쪼갤까"는 `feature-planning` 스킬(planner 담당)의 몫이다.

## 왜 PRD만 읽고 ARCH/ADR은 안 읽는가

`01_ARCHITECTURE.md`, `02_ADR.md`를 실제로 통독해 대조한 결과, 이 두 문서는 PRD의 Non-Goals/Constraints를 **인용**할 뿐 독자적인 제약을 새로 만들지 않는다. ARCH §2 "DB 없음"은 "로그인/찜/리뷰 Non-Goals이므로"라고 PRD를 근거로 든 파생 결과이고, ADR-0002("영속 데이터베이스 미사용")의 Context 첫 줄도 "PRD에서 로그인/찜하기/리뷰 등 개인화 기능을 Non-Goals로 확정했다"고 PRD를 인용한다. 즉 적합성(이 요청이 프로젝트 스코프에 맞는가) 판단은 PRD 하나로 충분하다 — ARCH/ADR까지 읽으면 뒤이어 호출되는 planner와 같은 문서를 중복해서 읽는 비용만 생긴다.

## 절차

1. **PRD 통독**: `docs/00_PRD.md` 전체를 읽는다. 선별 읽기 금지 — Non-Goals(§2)/Constraints(§7)가 문서 전체에 흩어져 있다.

2. **적합성 판정**: 요청 원문을 §2 Non-Goals("회원가입/로그인", "찜하기/리뷰 작성/평점 부여 등 개인화 기능", "별도 i18n 시스템")과 §7 Constraints(1인 개발, TMDB 무료 티어 rate limit, DB 없음)에 대조한다.
   - `pass`: 충돌 없음
   - `reject`: 충돌 있음 — 어떤 조항과 충돌했는지 구체적으로 남긴다

   **주의**: 이 판정은 "판정"이지 "결정"이 아니다. reject라고 해서 당신이 작업을 중단시키거나, pass라고 해서 당신이 진행을 확정짓지 않는다 — 그건 오케스트레이터와 사용자의 몫이다. 예를 들어 "TMDB 리뷰 API로 리뷰 열람 컴포넌트 만들기"는 PRD Non-Goal의 "리뷰 **작성**"과 다른 것(TMDB가 이미 가진 리뷰를 **읽기 전용으로 표시**)이므로 pass다 — Non-Goal 문구에 "리뷰"라는 단어가 있다고 기계적으로 reject하지 않는다. 문구가 아니라 실제로 배제하려는 행위(개인화 기능 구축)와 일치하는지를 본다.

3. **모호성 점검** (적합성 pass일 때만 진행 — reject면 이 단계는 의미 없다): 아래 세 유형을 체크한다.
   - **복수해석형**: 요청을 두 가지 이상으로 해석할 수 있고, 해석에 따라 산출물(만들어질 컴포넌트/동작)이 달라지는가?
   - **근거없음형**: PRD 어디에도 이 요청을 뒷받침하거나 언급하는 부분이 없는 완전 신규 영역인가? 이 경우 인터뷰로 넘어가기 전에 **2번(적합성 판정)으로 되돌아가 재검토**한다 — 근거가 없다는 건 모호성이 아니라 애초에 적합성 문제(Non-Goal에 안 걸렸을 뿐 스코프 밖일 수 있음)일 가능성이 있다.
   - **필수값누락형**: 요청 원문에 이 기능을 구현하는 데 반드시 필요한 파라미터/범위/동작분기가 빠져 있는가? (예: 페이지네이션 단위, 정렬 기준, 빈 상태 처리 등)

   세 유형 모두 해당 없으면 인터뷰 없이 곧장 5번(산출)으로 간다.

4. **인터뷰**: 모호한 점이 있으면 사용자에게 확인한다.
   - `grilling` 스킬이 있으면 그걸 사용한다. 원래 "완성된 플랜을 스트레스테스트"하는 용도로 만들어졌지만, "요구사항 공백을 메운다"는 목적을 프롬프트에 명시하면 그대로 재사용할 수 있다.
   - 없으면 이 규칙을 따른다:
     - 유형별로 나눠 순차 질문한다(한 번에 몰아묻지 않는다 — 앞선 답이 뒤 질문의 필요 여부를 바꿀 수 있다)
     - 복수해석형은 `AskUserQuestion`으로 옵션 2~4개를 제시한다 — 자유서술을 유도해 또 다른 모호함을 만들지 않는다
     - 질문 개수 상한은 없다. 매 라운드 끝나면 "이걸로 확정 가능한가"를 재판단하고, 가능하면 즉시 멈춘다 — 필요 이상으로 캐묻지 않는다

5. **산출**: 아래 고정 포맷으로 `_workspace/<브랜치명>/00_analysis_result.md`를 작성한다.

## 출력 포맷

```markdown
## 확정 요구사항
리뷰 열람 컴포넌트 — TMDB `/movie/{id}/reviews` API로 영화 상세 페이지에 리뷰 목록 표시. 페이지네이션 10개씩, 최신순 정렬. content 길면(200자 이상) 접어서 "더보기"로 표시. author_details.rating은 null이면 별점 영역 자체를 숨김.

## 적합성 판정
- 판정: pass
- 상세: 없음

## 관련 근거
- FR-ID: FR-3 (영화 상세 페이지)

## 인터뷰 요약
페이지네이션 단위/정렬기준 미기재 — 확인 결과 10개씩·최신순으로 확정. 긴 리뷰 처리 방식도 물었으며 200자 기준 접기로 정함.
```

필드는 넷뿐이다("확정 요구사항"/"적합성 판정"/"관련 근거"/"인터뷰 요약"). planner는 이 파일만 입력으로 받으며 PRD를 다시 읽지 않으므로, "확정 요구사항"에는 나중에 planner가 재해석할 필요가 없을 만큼 구체적으로 적는다.

## 재작업(적합성이슈) 처리

QA가 배치 구현 결과에서 뒤늦게 PRD Non-Goals/Constraints 위반을 발견해 "적합성이슈"로 리젝트하면, 오케스트레이터는 당신을 재호출하지 않는다 — 같은 PRD를 다시 읽어도 판정이 바뀔 이유가 없기 때문이다. 오케스트레이터가 바로 사용자에게 에스컬레이션한다.
