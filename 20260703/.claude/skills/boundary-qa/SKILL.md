# Boundary QA

TMDB 탐색 웹앱의 배치 단위 검증 절차. `qa` 에이전트가 이 절차를 따른다.

## 왜 "존재 확인"이 아니라 "교차 비교"인가

두 컴포넌트가 각각 개별적으로는 올바르게 구현돼 있어도, 연결 지점에서 계약이 어긋나는 경우가 실제 버그의 대다수다. TypeScript 제네릭 캐스팅이나 `any`가 끼면 컴파일러도 이걸 못 잡는다. "API가 존재하는가"와 "API 응답이 호출측 기대와 일치하는가"는 전혀 다른 검증이다 — 후자만이 실제 버그를 잡는다.

## 왜 배치 단위인가 (모듈 1개 끝날 때마다가 아니라)

이 프로젝트는 병렬 배치 안 태스크들이 worktree로 격리되어 동시에 개발된다. 배치 안 태스크 하나가 먼저 끝나도, 같이 병렬로 도는 다른 태스크가 아직 없으면 그것과의 경계면 비교는 반쪽짜리다. 배치 전원이 merge-back된 뒤 한 번에 검증해야 배치 내 태스크들끼리 온전히 교차 비교할 수 있다.

## 검증 대상 (이 프로젝트 기준)

| 검증 대상 | 왼쪽(생산자) | 오른쪽(소비자) |
|----------|-------------|---------------|
| tmdb-client 응답 shape | 함수 반환 타입 | 그걸 쓰는 Server Component/훅의 기대 타입 |
| 공용 UI 컴포넌트 props | `ui`의 `ScrollRail`/`ContentCard`/`AdultToggle` 등 정의 | `home`/`detail`/`search` 등 소비 측 실제 사용법 |
| 라우팅 | `src/app/` 하위 실제 page 경로 | 코드 내 `href`/`router.push`/`Link` 값 |
| Route Handler ↔ TanStack Query | `/api/*` 응답 shape | `useInfiniteQuery` 등 훅의 제네릭 타입, `.data` unwrap 여부 |
| 캐싱 정책 | `01_ARCHITECTURE.md` §8 (목록 `revalidate:3600`, 상세 `revalidate:86400`) | 실제 fetch의 `revalidate` 값 |
| 에러/엣지케이스 정책 | `01_ARCHITECTURE.md` 에러 정책 표 | 실제 `notFound()`/`error.tsx`/`retry` 설정 |

이 프로젝트는 로그인/개인화/DB가 Non-Goal이라 상태전이맵 검증은 대상이 아니다 — 억지로 체크하지 않는다. 다만 **구현이 실제로 이 Non-Goal을 위반하는 구조로 만들어졌는지**는 별도로 확인한다(아래 5번).

## 절차

1. 배치 내 모든 태스크명과 각각의 "근거"(`01_planner_tasks.md`)를 확인한다.
2. 위 표의 각 행에 대해 **양쪽 코드를 동시에 Read**한다 — 한쪽만 보고 판단하지 않는다.
3. 이전 배치와의 접점도 확인한다: `02_progress_state.md`를 읽어 이전 배치가 확정한 인터페이스를 파악하고, 이번 배치가 그걸 올바르게 소비하는지 본다.
4. `00_analysis_result.md`가 있으면 확인한다(analyst가 이미 적합성 판정을 끝낸 요청이면 정상적으로는 여기서 문제가 없어야 한다).
5. **적합성 위반 여부를 확인한다**: 구현 결과가 실제로 PRD Non-Goals(로그인/찜/리뷰작성/i18n 등)나 Constraints(DB 없음 등)를 위반하는 구조인지 본다 — analyst의 사전 판정을 통과했더라도, 구현 단계에서 스코프가 슬쩍 넘어가는 경우(예: 단순 표시 기능으로 시작했는데 서버 저장소가 필요한 구조로 구현됨)가 있다.
6. 판정한다. 리젝트라면 반드시 아래 중 하나로 분류한다:
   - **구현이슈**: 코드 실수(타입 불일치, 정책 미준수, props 오용) — implementer가 태스크 스코프 안에서 고칠 수 있음
   - **분해이슈**: planner의 의존성 판단 자체가 틀림(독립이라던 두 태스크가 실제로는 얽혀 충돌, 또는 근거 문서 해석 오류)
   - **적합성이슈**: 구현 결과가 PRD Non-Goals/Constraints를 위반함 — 코드를 고쳐서 될 문제가 아니라 스코프 자체를 사용자가 재확인해야 함. 이 유형은 재시도 루프에 태우지 않는다(오케스트레이터가 바로 사용자에게 에스컬레이션한다)
7. 아래 고정 포맷으로 반환한다.

```markdown
- QA 판정: 통과 / 리젝트
- 유형: 구현이슈 / 분해이슈 / 적합성이슈  (리젝트 시에만)
- 상세: home이 ScrollRail에 items prop을 배열로 넘기는데 컴포넌트는 { data, hasMore }를 기대함 (파일:라인)
```

한쪽 경계면 상대가 아직 없거나 접근 불가능하면 "미검증"으로 표시하고 이유를 남긴다 — 임의로 통과 처리하지 않는다. 실제 변경 파일은 브랜치의 git diff로 직접 확인한다(implementer의 자기 보고를 근거로 삼지 않는다).
