## T6 — 판정: PASS

검색 페이지 `/search` UI + 무한스크롤(FR-2) 통합 정합성 검증. 4개 경계면 모두 이상 없음.

### API↔훅 (일치)
- 생산자 `src/app/api/search/route.ts`: `NextResponse.json(data)`에서 `data`는 `searchMulti(...)` = `SearchResponse = Paginated<MultiSearchResult>`. 빈 query·에러 경로도 동일 shape 유지(`{ page, results, total_pages, total_results }`).
- 소비자 `use-search-infinite.ts:43`: `const data: SearchResponse = await response.json()`, `useInfiniteQuery` 페이지 타입 = `SearchResponse`. 래핑 불일치 없음(배열 아님, 페이지 래퍼 그대로).
- `search-explorer.tsx:139` `data.pages.flatMap((page) => page.results)` → `MultiSearchResult[]`. `getNextPageParam`은 `lastPage.page < lastPage.total_pages`로 페이지 필드 정확 소비.
- snake_case(TMDB 원본: `media_type`/`poster_path`/`release_date`/`first_air_date`/`vote_average`/`profile_path`/`known_for`)를 앱 타입에서도 그대로 유지 — 변환 누락/불일치 없음.

### media_type 분기 / 유니온 내로잉 (일치, as 단언 없음)
- `partitionResults()`(58~79): 판별 유니온 `MultiSearchResult`를 `media_type === "movie" | "tv" | else` 분기로 내로잉. `as` 없이 `MovieSearchResult[]`/`TVSearchResult[]`/`PersonSearchResult[]` 확정. tsc 통과가 내로잉 정당성 입증.
- `knownForLabel()`(82~92): `known_for`(`Array<MovieSearchResult|TVSearchResult>|undefined`)를 `media_type === "movie" ? item.title : item.name`로 내로잉, undefined/빈 배열 방어.

### 에러/엣지케이스 (§4·§3.2 준수)
- 초기(입력 전) `!hasSubmitted` → EmptyState(219). 로딩 `isLoading` → SkeletonCard 12장 그리드+`aria-busy`(224). 에러 `isError` → ErrorState + `refetch`(235, 훅 `retry:1` 이후 최종 실패). 무결과 `!hasResults` → EmptyState(243). 4상태 명확 분리.
- 무한스크롤 정지: sentinel `enabled = Boolean(hasNextPage) && !isFetchingNextPage`(127). 마지막 페이지 → `getNextPageParam` undefined → `hasNextPage` false → 옵저버 미부착 → 추가 UI 없이 조용히 정지(안내 문구 없음). 로딩 중 미부착으로 중복 요청 차단. `isFetchingNextPage`에만 Skeleton 6장 append(328).
- 429 패스스루: route가 `error.status(400~599)` 그대로 반환, 훅은 `retry:1`만(재시도 로직 추가 없음).
- 데이터 결측: `yearOf("")→null`(ContentCard "연도 미상"), poster/profile null → PosterImage/PersonAvatar 플레이스홀더, `known_for` 없으면 부제 줄 생략, 타입별 빈 섹션 `length>0` 가드로 숨김.
- aria-live: 214줄 `<div className="sr-only" role="status" aria-live="polite">{liveMessage}</div>` 존재·위치 정확. 로딩/N건/무결과/오류 변화 안내, 입력 전 침묵.

### SubmitEvent 교체 적절성 (오케스트레이터 수정분 재확인 — 적절)
- `@types/react`가 이 프로젝트에서 커스터마이징됨. `index.d.ts:2077-2083`에서 `FormEvent`가 `@deprecated FormEvent doesn't actually exist` 처리, 대체로 `SubmitEvent`/`ChangeEvent`/`InputEvent` 지목.
- `index.d.ts:2168` `interface SubmitEvent<T = Element> extends SyntheticEvent<T, NativeSubmitEvent>` — **제네릭 존재**. `SubmitEvent<HTMLFormElement>` 유효.
- `onSubmit?: SubmitEventHandler<T>`(2305) = `EventHandler<SubmitEvent<T>>`(2242). `<form>`의 `onSubmit`에 `handleSubmit(event: SubmitEvent<HTMLFormElement>)` 시그니처 정확 매칭.
- `handleSubmit` 내부 `event.preventDefault()`는 SyntheticEvent 상속 메서드 — 런타임/타입 모두 호환. tsc/eslint 0에러가 정합 입증. **교체 타당.**
