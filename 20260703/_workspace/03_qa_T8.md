## T8 — 판정: PASS

### client.ts 확장 정당성 (필독 판단)
- `01_ARCHITECTURE.md` §5.1은 "인터페이스: `getTrending()`, `getMovie(id)`, `getTvShow(id)`, `getPerson(id)`, `searchMulti(query)`, `getGenres(type)` **등**" — 예시 나열이며 "등"으로 비완전함을 스스로 명시. 반면 실제 완료 기준으로 쓰인 `01_planner_tasks.md` T2 섹션(65줄)은 구체적 함수 목록에 `getMovieCredits`/`getMovieRecommendations`는 포함하되 TV용 대응 함수(`getTvCredits`/`getTvRecommendations`)는 **나열하지 않음** — `getTvShow`, `getTvSeason`만 명시.
- T8 완료 기준(121줄)은 "출연진/추천 렌더"를 명시적으로 요구하므로 데이터 소스가 필수. T2 스펙 누락은 실질적(architecture doc이 아니라 planner 태스크 목록 기준)이며, movie 버전과 동일한 `tmdbRequest` 패턴(DETAIL revalidate, throw 기반 에러 전파, 404 미분리 — 하위 리소스라 root에서 이미 notFound 처리)을 그대로 미러링했다. 타입도 신규 정의 없이 기존 `Credits`/`Paginated<TVShow>` 재사용 — 과다 확장 아님.
- 판정: **T2 "미해결 재분류" 불필요.** T2가 커버 불가능했던 항목(TV 크레딧/추천)을 T8이 필요 시점에 확립된 패턴으로 최소 확장한 것으로, 아키텍처 일관성(단일 tmdb-client 경유, ADR-0003) 위반 없음. 다만 `01_ARCHITECTURE.md §5.1` 함수 예시 목록에 `getTvCredits`/`getTvRecommendations`를 사후 반영하는 문서 보강은 권장(선택, 강제 아님).

### API↔훅
- 해당 없음(T8은 Server Component 직접 호출, Route Handler/훅 없음) — 확인 결과 실제로 `src/app/tv/[id]/page.tsx` 어디에도 `fetch`/`useQuery` 없이 `tmdb-client` 함수만 호출. ADR-0003 패턴 준수.

### 에러/엣지케이스
- notFound 순서: `page.tsx:76-92` — id 파싱 실패(비정수/음수) 즉시 notFound → `getTvShow` null 체크 notFound → 이후에만 `getTvCredits`/`getTvRecommendations` 병렬 호출. T7(`movie/[id]/page.tsx:129-145`)과 완전히 동일 패턴. 없는 id가 error.tsx로 새지 않음 확인.
- `error.tsx`: `unstable_retry` 연결, `console.error` 로깅, movie 버전과 동일 구조.
- 결측 처리: 개요 없음→"개요 정보가 없습니다."(`page.tsx:157`), 방영일 없음→"방영일 미정"(`page.tsx:97`), 러닝타임 없음→"러닝타임 정보 없음"(`page.tsx:98-100`). 실제 dev 서버 렌더(`/tv/1399`, 왕좌의 게임)에서 "러닝타임 정보 없음" 대체 문구 실제 노출 확인(TMDB가 episode_run_time을 빈 배열로 반환하는 케이스 실증). cast/recommended 빈 배열 시 섹션 자체 숨김(`page.tsx:162`, `183` 조건부 렌더) — T7과 동일.
- 429/5xx: `tmdbRequest` 공용 함수 경유라 T2 정책 그대로 적용(패스스루/throw), 신규 함수도 동일 함수 사용.
