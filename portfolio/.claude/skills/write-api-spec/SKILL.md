# write-api-spec

api-spec agent의 Phase1 전담 스킬이자, backend agent가 병렬 개발 중 스펙을 실시간 갱신할 때도 직접 호출하는 **공유 자산**이다.

## Input
- `target`: 스펙 작성 대상 엔드포인트/도메인 식별자 (필수)
- `request_context`: 이 스펙이 왜 필요한지, 무엇을 요구하는지에 대한 원본 요청 원문 (필수 — 없으면 `target`만으론 어떤 엔드포인트/파라미터를 새로 만들거나 바꿔야 하는지 알 수 없음)
- `output_path`: 산출물 저장 경로 (생략 시 `docs/api/{target}.md`)

## 절차

1. **요구사항 확인** — `request_context`에서 이번에 추가/변경해야 하는 엔드포인트·파라미터·응답 필드가 무엇인지 파악
2. **지식 소스 확보** (키/시크릿 없이 확인 가능한 범위만):
   - `target`이 외부 시스템 프록시(예: 네이버 데이터랩)면 → 해당 시스템 공식 API 문서에서 요청/응답 필드·타입 확인
   - `target`이 내부 데이터 조회(예: Supabase 게시글 조회)면 → 실제 데이터 소스(DB 스키마 또는 원본 수집 대상)에서 실제 필드·타입 확인
   - 추측으로 필드/타입 확정 금지 — 확인 안 되면 "미확정" 표기 후 사용자 확인 요청
3. **기존 스펙과 재사용 관계 확인** — 데이터 소스가 겹치면 부분집합으로 참조, 중복 정의 금지
4. **계약 작성** (5개 필수): 엔드포인트 / 파라미터 / 응답타입(런타임검증 스키마로 옮길 수 있게) / 에러shape / 예시
5. **외부설정 필요사항 명시** — 키 이름/용도만(발급처·값은 사용자 몫)
6. **저장**: `output_path` 지정 없으면 `docs/api/{target}.md`

## 금지
- 키/시크릿 값 하드코딩
- 확인 없이 추측으로 타입 확정
- service/controller 구현 코드 작성 (spec-only는 문서만)
