# Feature Implementation

TMDB 탐색 웹앱의 개발 절차. `implementer` 에이전트가 이 절차를 따른다.

## 왜 스코프를 엄격히 지키는가

같은 배치의 다른 태스크는 이미 독립적이라고 planner가 판단한 것이다 — 그 경계를 넘어가면 병렬 실행의 전제(파일/인터페이스 안 겹침)가 깨지고, worktree 격리도 무의미해진다. 스코프 밖에서 뭔가 고쳐야 할 것 같으면, 고치지 말고 비고에 남긴다 — planner나 QA가 판단할 문제다.

## 절차

0. **작업 디렉토리 확인**: 프롬프트에 명시된 작업 디렉토리로 이동해, 참고/수정 대상으로 언급된 파일·폴더(`src/components/`, `src/hooks/` 등)가 실제로 존재하는지 먼저 확인한다. 없다면 worktree가 잘못된 기준(오래된 커밋)에서 만들어진 것이다 — 임의로 다른 경로(메인 체크아웃 등)를 건드리지 말고, 자신의 worktree 안에서 `git merge --ff-only {기준 브랜치}`로 자기 브랜치를 순수 전진시킨 뒤 다시 확인한다. 이 방법도 막히면(예: 이미 uncommitted 변경이 있음) 상태를 있는 그대로 보고하고 멈춘다.
1. **태스크 확인**: 담당 태스크의 "작업"과 "근거"(FR-ID/ARCH 섹션)를 읽는다. 근거가 가리키는 문서 부분을 직접 대조해 애매한 부분을 없앤다.
2. **파일 배치 판단**: 새 파일을 만들거나 옮길 때 `CLAUDE.md`의 "폴더 배치 판단" 원칙에 따라 대상 폴더 성격과 맞는지 확인한다. 네이밍 케이스(컴포넌트=PascalCase, 훅=camelCase+use접두사 등)도 `CLAUDE.md` 표를 따른다.
3. **구현**: `01_ARCHITECTURE.md`의 에러/엣지케이스 정책(404→`notFound()`, fetch 실패→`error.tsx`, 429→패스스루+`retry:1`, 데이터 결측→대체문구/섹션숨김)을 지킨다.
4. **결과 반환**: 아래 고정 포맷으로 오케스트레이터에게 반환한다.

```markdown
### {태스크명}
- 상태: 완료 / 실패
- 비고: {계획과 다르게 처리한 부분. 없으면 "없음"}
```

변경 파일 목록은 반환값에 넣지 않는다 — git diff가 이미 정확한 진실을 갖고 있고, 자기 보고는 코드와 어긋날 위험이 있다(스타일 문서가 실제 CSS와 갈라졌던 것과 같은 함정).

**`git push`를 하지 않는다.** 커밋까지만 하고 멈춘다 — push는 오케스트레이터도 하지 않는 작업이고(merge-back은 로컬 `git merge`로 처리), 사용자 승인 없는 원격 push는 하네스 전체의 금지 원칙이다. 과거 세션에서 이 지침 부재로 origin에 의도치 않은 브랜치/커밋이 올라간 사고가 있었다.

## 재작업 처리

QA가 "구현이슈"로 리젝트하면 구체적 지적사항만 고친다 — 지적받지 않은 부분을 임의로 리팩터링하지 않는다. 최대 2회까지 재작업하며, 그 이상은 오케스트레이터가 사용자에게 에스컬레이션한다.
