# Feature Planning

TMDB 탐색 웹앱의 계획 절차. `planner` 에이전트가 이 절차를 따른다. "확정된 요구사항을 어떻게 쪼갤까"만 다룬다 — "이 요구사항이 뭐고 타당한가"는 `feature-analysis` 스킬(analyst 담당)이 먼저 끝내고 넘겨준다.

## 왜 ARCH/ADR을 매번 전체 통독하는가

`01_ARCHITECTURE.md`(164줄), `02_ADR.md`(113줄) 합쳐 277줄 — 선별해서 읽어도 절약되는 비용보다, 모듈 경계·에러정책·결정근거를 놓쳐서 잘못된 계획을 세울 리스크가 더 크다. 예를 들어 "검색 API 재시도 로직 추가" 요청은 언뜻 UI와 무관해 보이지만 `02_ADR.md`의 rate limit 리스크 수용 근거와 직결된다 — 카테고리로 미리 걸러내면 이런 연결을 놓친다.

`docs/00_PRD.md`는 읽지 않는다. analyst가 이미 통독해 적합성 판정과 확정 요구사항으로 정리해 넘겨준다 — 같은 문서를 또 읽으면 중복 판독이다. `docs/03_DESIGN.md`도 읽지 않는다. 최초 base 스냅샷이고 실제 디자인 토큰은 `src/app/globals.css`로 이관되었다 — 읽으면 stale한 값을 근거로 쓰게 된다.

## 절차

1. **문서 통독**: `01_ARCHITECTURE.md` + `02_ADR.md` 전체를 읽는다. 스타일/디자인 관련 태스크면 `globals.css`도 읽는다.
2. **태스크 분해**: analyst의 확정 요구사항(`00_analysis_result.md`)을 작업 단위로 나눈다. 기준은 **의존성 유무**다 — "서로 파일/인터페이스를 공유하거나 순서가 있는가?"를 먼저 묻고, `01_ARCHITECTURE.md` §5의 모듈 경계(tmdb-client/home/search/detail/ui)는 참고 힌트로만 쓴다.

   **반례로 기준을 확인한다**: `detail` 모듈은 `/movie/[id]`, `/tv/[id]`, `/person/[id]` 세 라우트를 포함하지만 셋은 서로 안 얽힌다 — 모듈이 같다고 한 태스크로 묶으면 병렬 기회를 놓친다. 반대로 `home`이 `ui`의 신규 컴포넌트를 써야 한다면, 모듈이 달라도 순서 있는 하나의 의존관계로 묶는다.
3. **배치 그룹핑**: 서로 독립적인 태스크(파일/인터페이스 안 겹치고 순서 제약 없음)를 같은 배치에 넣는다. 의존관계가 있으면 다른 배치로 나눈다.
4. **산출**: 아래 고정 포맷으로 `_workspace/<브랜치명>/01_planner_tasks.md`를 작성한다.

## 출력 포맷

```markdown
## 배치1
### tmdb-client
- 작업: getTrendingAll() 함수 추가 — 히어로 캐러셀용 트렌딩 데이터 조회
- 근거: FR-1, ARCH §5.1

## 배치2 (배치1 완료 후)
### home
- 작업: HeroCarousel 컴포넌트 + 홈 페이지 데이터 페칭 연결
- 근거: FR-1

### search
- 작업: 통합 검색 페이지 + /api/search Route Handler
- 근거: FR-2, ARCH §5.3

## 미정 사항
- search 모듈에 신규 필터 UI를 붙일 때 `ui`의 `GenreFilter`를 재사용할지 새로 만들지 ARCH §5.3/§5.5만으로는 판단 불가 — 사용자 확인 필요
```

필드는 "작업"과 "근거" 둘뿐이다. 우선순위·예상 소요시간·정확한 파일 경로는 넣지 않는다 — 배치 순서가 이미 실행 순서고, 파일 배치는 implementer가 `CLAUDE.md` 폴더 배치 판단 원칙으로 그때그때 정한다.

## 재작업(분해이슈) 처리

QA가 "분해이슈"로 리젝트하면, 문제 상황 설명과 함께 재호출된다. **영향받은 배치만** 다시 나눈다 — 이미 QA를 통과한 이전 배치는 재검토하지 않는다. 국소적 판단 오류가 문서 전체 재해석을 요구하는 경우는 드물다.
